# W13：写操作边界：确认、幂等与审计

阶段：阶段 3：Tool / Agent / MCP

## 为什么这周适合你

一旦工具有写操作，风险就完全不同。删除、修改、发送、创建都必须有确认、幂等和审计。

你是有经验的前端，所以本周不会停留在概念解释，而是从“一个真实 AI 功能如何进入现有前端系统”开始。

## 本周要构建什么

实现一个模拟写工具 createDevTask，只允许生成待确认任务，不允许模型直接提交最终结果。

## 前端切入点

前端必须出现二次确认面板：展示将要执行的动作、参数、影响范围和取消按钮。

## 本周必须补的后端基础

- 写操作权限
- 幂等键
- 审计日志
- 二次确认 token
- 副作用隔离

## 本周核心能力

- Write Tool
- Human Confirmation
- Idempotency
- Audit Log
- 副作用边界

## 一周节奏（每天 2～3 小时）

- [Day 01：真实场景与边界](../lessons/week-13/day-01.md) — 先判断这个能力在真实产品里解决什么问题，以及前端、服务端、模型分别负责什么。
- [Day 02：最小代码闭环](../lessons/week-13/day-02.md) — 只写一条能跑通的端到端链路，优先把输入、服务端、模型或数据、输出串起来。
- [Day 03：后端补点与可靠性](../lessons/week-13/day-03.md) — 补上这周必须懂的后端基础，并处理超时、校验、权限、日志或失败兜底。
- [Day 04：失败样本与评测](../lessons/week-13/day-04.md) — 不要只看成功演示，构造失败样本，把问题归因到数据、模型、代码或产品边界。
- [Day 05：作品集与面试表达](../lessons/week-13/day-05.md) — 整理架构图、关键代码、日志、评测结果和一段能被面试追问的讲法。

## 本周代码任务

- [ ] 实现 createDevTaskDraft，只生成草稿。
- [ ] 实现 confirmDevTask，必须携带 confirmationId。
- [ ] 记录 audit log：谁、何时、确认了什么。

## 本周交付证据

- [ ] 写操作确认流程截图
- [ ] 重复提交防护日志
- [ ] 审计日志样例

## 面试讲法

我会把只读工具和写工具分开，写操作必须用户确认、幂等防重复，并留下审计日志。

## 通过标准

- 能跑出端到端最小闭环；
- 能说清前端、服务端、模型、数据各自边界；
- 有失败样本和处理策略；
- 有日志、评测、截图、Trace 或架构图证据；
- 能把本周能力写进作品集项目，而不是只背概念。
