# W05：会话、日志、成本：AI 功能上线前必须补的后端基础

阶段：阶段 1：AI 应用工程底座

## 为什么这周适合你

AI 功能上线后，最常见的问题是慢、贵、偶发失败、无法复盘。你需要能保存调用记录并定位问题。

你是有经验的前端，所以本周不会停留在概念解释，而是从“一个真实 AI 功能如何进入现有前端系统”开始。

## 本周要构建什么

为前 4 周能力加调用日志：requestId、userInput 摘要、provider、model、耗时、token、错误类型、promptVersion。

## 前端切入点

前端展示 requestId，用户反馈问题时能把 requestId 给到开发定位。

## 本周必须补的后端基础

- 日志模型设计
- 简单持久化 JSON / SQLite 思路
- token 与成本字段
- 敏感信息脱敏
- 请求追踪

## 本周核心能力

- 调用日志
- Token Usage
- Cost Tracking
- 脱敏
- 问题复盘

## 一周节奏（每天 2～3 小时）

- [Day 01：真实场景与边界](../lessons/week-05/day-01.md) — 先判断这个能力在真实产品里解决什么问题，以及前端、服务端、模型分别负责什么。
- [Day 02：最小代码闭环](../lessons/week-05/day-02.md) — 只写一条能跑通的端到端链路，优先把输入、服务端、模型或数据、输出串起来。
- [Day 03：后端补点与可靠性](../lessons/week-05/day-03.md) — 补上这周必须懂的后端基础，并处理超时、校验、权限、日志或失败兜底。
- [Day 04：失败样本与评测](../lessons/week-05/day-04.md) — 不要只看成功演示，构造失败样本，把问题归因到数据、模型、代码或产品边界。
- [Day 05：作品集与面试表达](../lessons/week-05/day-05.md) — 整理架构图、关键代码、日志、评测结果和一段能被面试追问的讲法。

## 本周代码任务

- [ ] 设计 ai_call_logs 数据结构，不保存完整敏感输入。
- [ ] 每次模型调用都写入日志，失败也要写。
- [ ] 做一个简单日志查看页面或命令，按 requestId 查询。

## 本周交付证据

- [ ] 调用日志样例
- [ ] 一次失败 requestId 追踪记录
- [ ] 成本 / 延迟统计截图或表格

## 面试讲法

我不只看接口 200，还会记录模型、耗时、token、成本、错误类型和质量样本，用 requestId 复盘问题。

## 通过标准

- 能跑出端到端最小闭环；
- 能说清前端、服务端、模型、数据各自边界；
- 有失败样本和处理策略；
- 有日志、评测、截图、Trace 或架构图证据；
- 能把本周能力写进作品集项目，而不是只背概念。
