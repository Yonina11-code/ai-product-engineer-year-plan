# W03：结构化输出：让模型结果能进入业务系统

阶段：阶段 1：AI 应用工程底座

## 为什么这周适合你

模型输出一段自然语言很容易，但业务系统需要稳定字段。你要解决的是格式错误、字段缺失、内容不合格和自动修复。

你是有经验的前端，所以本周不会停留在概念解释，而是从“一个真实 AI 功能如何进入现有前端系统”开始。

## 本周要构建什么

为一个职场沟通改写功能设计 JSON 输出契约，服务端用 Schema 校验，不合格时自动 repair 一次，仍失败就降级。

## 前端切入点

前端只展示已经校验过的结果，不直接相信模型原文；错误状态要能告诉用户是格式失败还是内容失败。

## 本周必须补的后端基础

- 运行时 Schema 校验
- JSON parse 防御
- 错误码设计
- 一次重试策略
- 服务端字段契约

## 本周核心能力

- 结构化输出
- Schema 校验
- Repair Prompt
- 业务字段契约
- 降级策略

## 一周节奏（每天 2～3 小时）

- [Day 01：真实场景与边界](../lessons/week-03/day-01.md) — 先判断这个能力在真实产品里解决什么问题，以及前端、服务端、模型分别负责什么。
- [Day 02：最小代码闭环](../lessons/week-03/day-02.md) — 只写一条能跑通的端到端链路，优先把输入、服务端、模型或数据、输出串起来。
- [Day 03：后端补点与可靠性](../lessons/week-03/day-03.md) — 补上这周必须懂的后端基础，并处理超时、校验、权限、日志或失败兜底。
- [Day 04：失败样本与评测](../lessons/week-03/day-04.md) — 不要只看成功演示，构造失败样本，把问题归因到数据、模型、代码或产品边界。
- [Day 05：作品集与面试表达](../lessons/week-03/day-05.md) — 整理架构图、关键代码、日志、评测结果和一段能被面试追问的讲法。

## 本周代码任务

- [ ] 定义 rewriteResult schema：rewrittenText、tone、riskFlags、reason。
- [ ] 服务端校验模型 JSON，失败后只 repair 一次。
- [ ] 前端分别展示成功、格式失败、内容不合格和降级结果。

## 本周交付证据

- [ ] Schema 文件或类型定义
- [ ] 10 条格式失败样本
- [ ] 修复前后对照日志

## 面试讲法

我区分了格式正确、内容合格和业务成功，模型结果必须先过 Schema 和业务校验才能进入页面。

## 通过标准

- 能跑出端到端最小闭环；
- 能说清前端、服务端、模型、数据各自边界；
- 有失败样本和处理策略；
- 有日志、评测、截图、Trace 或架构图证据；
- 能把本周能力写进作品集项目，而不是只背概念。
