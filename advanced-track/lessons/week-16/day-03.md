# W16 Day 03：可靠性增强

本周主题：MCP Server：把现有能力封装给 AI 使用

建议时间：2～3 小时

## 今天的目标

补错误处理、结构校验、边界条件和可观测信息。

## 今天不学什么

- 不学概念名词堆砌；
- 不重复你已经具备的基础前端知识；
- 不做和求职项目无关的玩具功能。

## 任务背景

MCP 的价值不是名词，而是把工具、资源和权限以标准方式暴露给模型客户端。

本周构建目标：

> 用 Node 写一个最小 MCP Server，封装一个只读接口契约查询工具。

## 今天要完成

- [ ] 补充校验、日志、异常处理和用户可见反馈。
- [ ] 把魔法字符串、散落 Prompt 或临时逻辑收敛到可维护位置。
- [ ] 记录一次失败链路，不只记录成功链路。

## 工程约束

- 每次只扩展一个明确能力；
- 真实模型、RAG、Agent、MCP 相关能力必须留下可验证证据；
- 不确定的接口字段、数据类型、业务含义不能猜；
- 所有 AI 输出都要考虑失败、空响应、格式错误和内容不合格；
- 结论优先沉淀通用规律，不把单个案例当成全部知识。

## 产出物

- 最小 MCP Server
- 工具调用截图
- 错误参数拒绝记录

## 今日复盘问题

1. 今天这项能力解决了真实 AI 项目里的什么问题？
2. 它和普通前端 CRUD 最大的差异在哪里？
3. 如果模型、检索或工具调用失败，用户会看到什么？
4. 这项能力能怎样写进你的求职项目？
5. 面试官追问时，你准备用哪个证据回答？
