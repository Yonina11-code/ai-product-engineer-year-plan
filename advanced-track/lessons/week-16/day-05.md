# W16 Day 05：作品集与面试表达

本周主题：项目三：MCP + RAG + Tool 综合接口助手

建议时间：2～3 小时

## 今天的目标

整理架构图、关键代码、日志、评测结果和一段能被面试追问的讲法。

## 今天不学什么

- 不学概念名词堆砌；
- 不重复基础前端知识；
- 不为了炫技引入暂时用不上的框架；
- 不做和求职项目无关的玩具功能。

## 真实任务背景

这一周把 RAG、Tool、Agent 和 MCP 串成一个真正能讲清楚的综合项目。

本周构建目标：

> 让助手通过 RAG 找证据，通过 Tool 查询结构化字段，通过受限 Agent 决定是否继续查，并把工具封装成 MCP。

## 前端怎么入手

前端重点展示完整 Trace：检索了什么、调用了什么工具、为什么拒答或继续查。

## 今天要完成

- [ ] 整理本周架构图、关键代码、日志、截图和评测证据。
- [ ] 写一段 30 秒版本和 3 分钟版本的面试讲法。
- [ ] 用自己的项目表达这句话：我能讲清一个 AI 系统里模型、RAG、Tool、Agent、MCP 分别解决什么问题，以及为什么不混用。
- [ ] 明确下周要继承的代码和要还的技术债。

## 今天要补的后端知识

- 多能力编排
- Trace 数据结构
- 错误传播
- 权限分层
- 综合项目验收

> 只补到能完成今天任务，不要求一次学成后端工程师。

## 工程约束

- 每次只扩展一个明确能力；
- 前端不直接持有模型 API Key；
- 服务端要处理失败、空响应、格式错误、超时和非法参数；
- 不确定的接口字段、数据类型、业务含义不能猜；
- 涉及 RAG、Tool、Agent、MCP 的能力必须留下 Trace 或可验证证据；
- 结论优先沉淀通用规律，不把单个 Demo 案例当成全部知识。

## 学习问答记录

### Q1：我学完今天后，最低通过标准是什么？

直接结论：不是“看懂概念”，而是能跑出或画出一条可验证链路，并能解释失败时怎么处理。

工程解释：AI 应用和普通 CRUD 最大的差异在于不确定性。模型可能慢、错、空、格式乱、成本高，所以每一天都必须留下运行证据或失败证据。

对应知识点：端到端闭环、服务端边界、失败兜底、日志 / Trace、面试证据。
