## Q2：这个 Demo 的前端和传统前端没有区别，真正的区别是不是后面用 API Key 调大模型？

### 直接结论

基本正确。

Vue 页面仍然负责传统前端工作：

```text
收集输入 → 调用接口 → 管理加载状态 → 展示结果或错误
```

AI 应用的主要变化发生在服务端：

```text
接收用户输入
→ 组织 Prompt
→ 调用真实大模型
→ 校验不稳定的模型输出
→ 返回可供前端使用的数据
```

### 当前写死的是哪里？

当前实验中的 `mock-model.js` 是模拟模型。它直接返回固定内容：

```js
return {
  rewrittenText: '固定的模拟结果',
  explanation: '固定的模拟说明',
}
```

后续会把：

```text
callMockModel()
```

替换为类似：

```text
callRealModel()
```

真实模型调用会使用 API Key，但 API Key 必须保存在服务端环境变量中，不能放在 Vue 前端。

### 不只是“换一个 API”

如果只把模拟返回换成模型接口，还不算完整的 AI 产品。真实场景还需要处理：

- Prompt 设计；
- 模型选择；
- 输出格式；
- 超时和重试；
- 空结果；
- 内容安全；
- Token 和费用；
- 日志和评测。

### 当前状态

真实 API Key 接入：**待后续实验**。

### 对应知识点

- AI 应用架构；
- 服务端密钥安全；
- 模型输出可靠性。

## Q3：Prompt 是不是就是页面选择的“专业语气”和“沟通对象”？

### 直接结论

不是。页面选择项只是完整 Prompt 中的动态上下文。

当前 Prompt 由四部分组成：

### 1. 固定任务

```text
你是一名职场沟通助手。
将用户原话改写为专业、清晰、坚定的工作沟通表达。
```

### 2. 页面提供的动态上下文

```text
沟通对象：产品经理
期望语气：坚定但不对抗
```

### 3. 产品固定约束

```text
不改变原意。
不虚构事实。
不增加用户没有承诺过的内容。
不要过度道歉。
```

### 4. 用户输入

```text
这个需求根本做不了，你们先想清楚再说。
```

可以暂时记成：

```text
完整 Prompt
= 固定任务
+ 动态上下文
+ 产品约束
+ 用户输入
+ 输出格式
```

### 对应知识点

- Prompt 模板；
- 静态指令；
- 动态上下文；
- 用户输入。

## Q4：用户每次输入的内容不同，真实场景中的 Prompt 都是一样的吗？

### 直接结论

同一个任务通常复用同一个 Prompt 骨架，但其中的用户输入和上下文会变化。

例如用户都在使用“消息改写”功能：

```text
固定：角色、任务、约束、输出格式
动态：原话、沟通对象、语气、沟通目的
```

因此两次完整 Prompt 不完全相同，但它们来自同一个模板。

### 什么时候需要不同 Prompt？

当任务发生变化时，应该使用不同 Prompt：

```text
改写消息 → rewritePrompt
分析语气 → analyzeTonePrompt
生成回复 → replyPrompt
总结会议 → summaryPrompt
```

真实产品可以通过明确按钮确定任务：

```text
[帮我改写] [分析语气] [帮我回复]
```

这种方式比让模型猜测用户意图更稳定，也更容易测试。

### 信息不足怎么办？

不能让模型自行补全事实。

例如用户只输入：

```text
这个方案我不同意。
```

系统可能缺少不同意的原因、沟通对象和用户目标。更可靠的流程是：

```text
检查必要信息
├─ 信息足够 → 构造 Prompt → 调用模型
└─ 信息不足 → 向用户追问一个明确问题
```

### 本题核心结论

> 同一任务复用 Prompt 骨架；任务变化时选择不同 Prompt；信息不足时先追问，不让模型猜事实。

### 后续会学到

- Prompt 路由；
- 上下文管理；
- 意图识别；
- 必填信息检查；
- Agent 决策边界。

## 本周阶段性理解

截至目前，可以把一个 AI 应用理解成：

```text
传统前端交互
  +
服务端 Prompt 组织
  +
模型调用
  +
不可靠输出的校验与产品化处理
```

当前最重要的认识不是“怎样写一个万能 Prompt”，而是：

1. Prompt 是一份完整任务说明，不等于用户输入；
2. 同一个产品可能拥有多个 Prompt 模板；
3. 模型不能替代明确的业务规则和必要信息校验；
4. AI 前端仍然需要扎实的传统前端能力。

## Q5：不同任务应该使用不同 Prompt，这个任务是服务器判断，还是由大模型判断？

### 直接结论

两种方式都可以，但应按照确定性优先：

1. 页面已经明确任务时，前端传递任务类型，服务端按规则选择 Prompt；
2. 可以通过普通程序规则确定时，由服务端判断；
3. 只有面对无法用固定规则覆盖的自由文本时，才让大模型识别意图；
4. 高风险或判断不明确时，向用户确认，不能让模型擅自决定。

### 方式一：用户在页面明确选择

例如页面提供：

```text
[帮我改写] [分析语气] [帮我回复]
```

用户点击“分析语气”后，前端提交：

```json
{
  "taskType": "analyzeTone",
  "content": "用户输入的内容"
}
```

服务端根据明确字段选择 Prompt：

```js
const promptBuilders = {
  rewrite: buildRewritePrompt,
  analyzeTone: buildAnalyzeTonePrompt,
  reply: buildReplyPrompt,
}

const buildPrompt = promptBuilders[taskType]
```

这是最稳定、最容易测试、成本最低的方式。

### 方式二：服务端通过普通规则判断

如果接口、路由或业务状态已经能说明任务，也不需要调用大模型判断。

例如：

```text
POST /api/rewrite       → 使用改写 Prompt
POST /api/analyze-tone  → 使用语气分析 Prompt
POST /api/reply         → 使用回复 Prompt
```

这里的“判断”实际上由服务端路由和业务代码完成。

### 方式三：让大模型识别用户意图

如果产品只有一个自由输入框，用户可能输入：

```text
这句话是不是太冲了？顺便帮我改得委婉一点。
```

普通程序不容易稳定判断。此时可以先让模型输出结构化意图：

```json
{
  "intents": ["analyzeTone", "rewrite"],
  "confidence": 0.86,
  "missingInformation": []
}
```

服务端再根据识别结果执行对应流程。

需要注意：模型的判断也可能错误，因此不能直接无条件相信。

### 推荐的真实产品流程

```text
用户明确选择任务
  → 服务端直接选择 Prompt

用户没有选择，但普通规则可以判断
  → 服务端规则选择 Prompt

用户输入是开放、复杂的自然语言
  → 大模型识别意图
  → 服务端校验识别结果

结果不明确或操作风险较高
  → 询问用户确认
```

### 为什么第一个项目不应该先使用大模型路由？

- 会增加一次模型调用和费用；
- 会增加响应时间；
- 意图识别本身也可能出错；
- 调试时难以判断是 Prompt 路由错了，还是业务 Prompt 错了；
- 当前页面已经有明确按钮和选项，没有必要增加不确定性。

### 本题核心结论

> 能由页面和程序明确确定的任务，不交给大模型猜；只有开放意图确实无法用规则判断时，才使用模型分类，并保留服务端校验和用户确认。

### 当前项目建议

职场沟通助手第一版使用：

```text
页面明确功能 → 前端传 taskType → 服务端选择 Prompt
```

等掌握结构化输出和评测后，再增加“自由输入自动识别任务”的实验。

### 对应知识点

- Prompt Router；
- 意图识别；
- Workflow；
- 结构化输出；
- 确定性规则与模型决策边界。

## Q6：怎样把 Mock 模型替换为 DeepSeek V4 Flash？

### 直接结论

API Key 只配置在服务端 `.env`，Vue 前端只提交业务参数和模型来源。

当前工程已经支持：

```text
教学 Mock ↔ DeepSeek deepseek-v4-flash
```

详细步骤：

[DeepSeek V4 Flash 接入说明](../docs/deepseek-api-setup.md)

### 调用链路

```text
Vue 选择 DeepSeek
→ 服务端读取 DEEPSEEK_API_KEY
→ 构造 messages
→ POST https://api.deepseek.com/chat/completions
→ 解析 JSON
→ 校验业务字段
→ 返回 Vue
```

### 为什么不能把 Key 传给 Vue？

- 浏览器代码和请求可以被用户查看；
- Key 泄露后可能产生未经授权的费用；
- 服务端无法可靠控制调用权限和频率；
- Prompt、日志、费用统计和结果校验应统一由服务端管理。

### 当前使用的模型

```text
deepseek-v4-flash
```

这是截至 2026-06-23 DeepSeek 官方文档列出的有效模型标识。

### 对应知识点

- 服务端环境变量；
- Bearer 鉴权；
- 模型适配层；
- JSON Output；
- Token 用量与成本。
