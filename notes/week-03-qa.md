# 第3周学习问答

主题：上下文、Token、对话记忆与 AI BI 产品拆解  
记录日期：2026-06-26

## Q1：SmartBI 这种产品打开很像 ChatGPT，看到案例时很懵，不知道它能帮我们公司的实际业务做什么。

### 直接结论

这类产品不能先理解成“聊天机器人”，而应该理解成：

```text
自然语言入口
+ 公司已有数据和业务系统
+ 可执行工具
+ 分析报告生成器
```

如果只看聊天框，会觉得它和 ChatGPT 很像；如果看它背后的工程链路，它真正的价值是把“业务人员想问的问题”转成“系统可以执行的数据分析流程”。

### 工程解释

以当前观察到的 SmartBI 页面为例，它不是只回答一段文字，而是围绕一个业务问题完成了这些步骤：

```text
用户问题
→ 选择智能体能力，例如数据洞察、SQL 查询、快速分析
→ 读取数据模型或业务语义
→ 生成查询或分析计划
→ 执行取数
→ 运行分析脚本
→ 生成图表
→ 输出结构化报告
→ 展示过程、耗时和产物
```

所以，判断它能不能帮公司业务，不应该先问“它会不会聊天”，而应该问：

1. 公司有没有高频、重复、依赖数据的业务问题；
2. 这些问题现在是不是靠人工查表、导 Excel、写 SQL、做 PPT；
3. 数据是否有明确口径，例如字段、指标、维度、时间范围；
4. 输出是否有固定交付形态，例如日报、周报、异常分析、经营看板、客户画像、库存分析、工单归因。

适合优先落地的场景通常长这样：

```text
每周都有人问；
答案依赖公司内部数据；
人工处理有固定步骤；
结果需要表格、图表或结论；
错误成本可控，允许人工复核。
```

不适合第一阶段落地的场景：

```text
问题一次性很强；
数据口径混乱；
结果直接影响高风险决策；
没有人能验证答案；
业务系统没有接口或数据出口。
```

### 对应知识点

- AI 产品问题定义；
- 上下文策略；
- 工具调用；
- 数据模型语义层；
- 可观测性；
- 业务场景优先级。

### 待实验

选一个公司真实业务问题，按下面模板拆解：

| 项目 | 记录 |
| --- | --- |
| 业务问题 | 例如：为什么本周某类订单下降 |
| 当前人工流程 | 查哪些系统、导哪些表、找谁确认 |
| 需要的数据 | 指标、维度、时间范围 |
| 可执行工具 | SQL、接口、Excel、BI 报表 |
| 期望输出 | 表格、图表、结论、建议 |
| 人工复核点 | 哪些结论必须人工确认 |

## Q2：它是不是只做 RAG？如果要落地，我们要怎么“喂养”它，它才会帮我们分析、生图、总结？

### 直接结论

不是只做 RAG。

RAG 只是其中一类能力，主要解决“从文档或知识库里找依据”的问题。SmartBI 这类 AI BI / Agent 产品通常是多种能力组合：

```text
RAG
+ 数据模型语义层
+ 工具调用
+ SQL / API / BI 查询
+ Python 或图表生成
+ Prompt 模板
+ 工作流状态
+ 结果校验
+ 权限和日志
```

如果只给它喂文档，它最多更会“解释”；如果要它会“分析、生图、总结”，必须喂给它可执行的数据、工具和规则。

### 工程解释

可以把“喂养 AI 产品”分成五层，不是一次性把资料丢进去。

#### 1. 喂业务语义

让系统知道公司里每个词是什么意思。

```text
销售额是什么；
订单量是什么；
有效客户怎么定义；
退款率怎么算；
部门、区域、产品线有哪些枚举；
同一个字段在不同系统里叫什么。
```

这一层不是训练模型，而是整理成数据字典、指标口径、字段说明和业务规则。

#### 2. 喂可查询数据

让系统能拿到真实数据。

```text
数据库；
BI 数据模型；
接口；
Excel；
CSV；
数据仓库；
已有报表。
```

如果只接文档，不接数据，它无法真正分析公司经营情况。

#### 3. 喂工具能力

让模型不能只“想”，还可以“做”。

```text
查询某个数据模型；
执行受限 SQL；
读取指定报表；
生成图表；
导出文件；
调用公司内部接口；
运行固定分析脚本。
```

这里的重点是工具契约明确：参数是什么、返回什么、失败怎么处理、权限怎么限制。

#### 4. 喂输出模板

让它知道不同任务应该交付什么。

```text
经营日报模板；
异常归因模板；
竞品分析模板；
销售复盘模板；
周报总结模板；
图表标题和指标说明规范。
```

这对应 Prompt Engineering，不是让模型自由发挥，而是让它按公司认可的结构输出。

#### 5. 喂测试集和反馈

让系统能持续变好。

```text
20～50 个真实问题；
每个问题的期望答案；
正确引用或数据来源；
失败案例；
人工评分；
用户反馈。
```

没有测试集，就只能靠感觉判断“好像变聪明了”，这不算工程落地。

### RAG 在其中的位置

RAG 适合回答：

```text
制度怎么规定；
接口字段是什么意思；
这个指标的口径是什么；
某份文档里有没有提到某个流程；
历史方案是怎么写的。
```

RAG 不擅长单独完成：

```text
实时销售趋势分析；
跨表聚合计算；
自动生成图表；
异常归因；
多步骤查询和总结；
根据结果继续追问取数。
```

这些需要工具调用和工作流。

### 一个更接近落地的架构

```text
用户问题
→ 意图识别
→ 选择业务场景
→ 检索相关口径和字段说明
→ 选择数据模型或工具
→ 生成查询计划
→ 执行查询
→ 校验结果
→ 生成表格/图表/总结
→ 展示来源、耗时、风险和可追问入口
```

这里 RAG 只负责“找依据和上下文”，不是全部。

### 第一阶段不要做太大

建议先做一个最小闭环：

```text
固定一个业务场景
固定一个数据源
固定三类问题
固定一种报告模板
只允许只读查询
所有结论显示数据来源
失败时明确告诉用户缺什么
```

例如：

```text
场景：每周经营数据复盘
数据：一张脱敏订单表
问题：销售额变化、TOP 产品、异常下降原因
输出：表格 + 图表 + 三条结论
复核：展示查询条件、时间范围和字段来源
```

### 对应知识点

- RAG 与 Agent 的区别；
- 语义层和数据字典；
- Function Calling / 工具调用；
- Workflow；
- Prompt 模板管理；
- 结构化输出；
- 可观测性和评测集；
- 企业权限边界。

### 待实验

先不要训练模型，也不要急着接所有系统。准备一个脱敏 CSV 或 mock 数据表，做一个最小 AI BI 流程：

```text
输入：一句自然语言问题
程序：只允许选择一个固定数据表
模型：生成分析计划和总结
工具：程序执行固定查询或聚合
输出：表格、图表配置、结论、风险提示
评测：至少 5 个问题，记录是否答对、是否引用了正确字段
```

验收重点：

1. 模型有没有编造字段；
2. 数据时间范围有没有展示；
3. 图表是否来自真实查询结果；
4. 总结是否超出数据支持；
5. 失败时是否能明确说“不知道缺什么”。

## 阶段性结论

SmartBI 这类产品的学习价值在于它把 AI 从“聊天”推进到“业务工作流”：

```text
ChatGPT 式界面只是入口；
RAG 只是找资料；
工具调用才让它能干活；
数据语义层决定它懂不懂公司业务；
评测和日志决定它能不能上线。
```

当前第 3 周可以先把它作为上下文策略案例，不急着完整复刻。后续学习到流式输出、Prompt 模板、工具调用、Agent 和 RAG 时，再逐步把这个案例扩展成一个小型 AI BI Agent。
