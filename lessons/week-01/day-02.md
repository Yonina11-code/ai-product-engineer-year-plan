## 1. 大模型不是答案数据库

传统数据库查询更接近：

```text
查询条件
→ 在已有数据中匹配
→ 返回符合条件的记录
```

大模型生成更接近：

```text
读取当前上下文
→ 计算接下来各种内容出现的可能性
→ 按生成策略逐步选择内容
→ 得到完整回答
```

它不是从数据库中找到一条预先存在的“标准回答”。

因此同一个问题可能：

- 使用不同措辞；
- 强调不同重点；
- 有时简洁，有时展开；
- 有时增加用户没有提供的信息；
- 有时生成听起来合理、实际没有依据的内容。

## 2. 什么是不确定性

不确定性并不等于随机胡说。

例如下面三句话都可能是合格结果：

```text
当前需求在现有条件下存在实现困难，建议先明确需求范围。
```

```text
这个需求目前还有一些实现条件需要确认，建议先对齐需求边界。
```

```text
基于当前信息，该需求暂时难以评估，建议先补充关键约束。
```

它们表述不同，但都可能满足任务。

真正需要控制的不是“每次必须一字不差”，而是结果是否满足产品标准。

## 3. Temperature 控制什么

可以先建立这个工程直觉：

| Temperature | 常见倾向 | 适合场景 |
| --- | --- | --- |
| 较低 | 更集中、保守、相似 | 字段提取、分类、格式化 |
| 中等 | 稳定与多样之间平衡 | 对话、改写、一般问答 |
| 较高 | 更多样、更有创造性 | 创意文案、头脑风暴 |

需要特别注意：

> Temperature 不是“质量”旋钮，也不是“真实性”旋钮。

把 Temperature 调低：

- 可能减少输出差异；
- 不能保证事实正确；
- 不能修复错误 Prompt；
- 不能补齐缺失的业务上下文；
- 不能替代结果校验。

## 4. Demo 实验

```text

```

这个实验是可控模拟器，不是真实模型。它的目的不是证明某个具体数值必然产生某种结果，而是帮助你观察三种生成倾向。

### 实验 A：低 Temperature

设置为：

```text

```

连续生成两轮，观察：

- 三个结果是否更接近；
- 是否仍然存在措辞差异；
- 低 Temperature 是否等于输出完全相同。

### 实验 B：高 Temperature

设置为：

```text
0.9
```

连续生成两轮，观察：

- 表达方式是否更丰富；
- 是否更容易增加原文没有的信息；
- “更有创意”是否一定“更适合产品”。

## 5. 什么是幻觉

在当前场景中，以下内容都可能属于没有依据的信息：

```text
本周五前补齐业务规则。
```

```text
我们将在下周一提供新方案。
```

```text
从当前排期来看……
```

用户并没有提供时间、承诺或排期，但模型可能生成得很自然。

幻觉最危险的地方不是语句不通顺，而是：

> 它听起来合理，让人不容易立即发现没有依据。

## 6. 不要用“好不好听”评测

职场改写可以先使用四个标准：

| 维度 | 判断问题 |
| --- | --- |
| 原意保持 | 是否仍然表达需求存在实现困难？ |
| 事实可靠 | 是否增加了原因、时间、排期或承诺？ |
| 语气匹配 | 是否专业坚定，但不过度攻击或示弱？ |
| 可直接使用 | 是否可以在真实工作场景中发送？ |

同一个输出可能：

- 语气很好，但虚构了承诺；
- 事实正确，但语气仍然攻击性强；
- 格式正确，但没有完成改写任务；
- 文案漂亮，但不适合用户选择的沟通对象。

所以 AI 评测通常是多维度的。

## 7. 和前端测试有什么相似与不同

相似点：

- 都需要明确输入和预期；
- 都要覆盖边界情况；
- 都不能只测试一次成功路径；
- 修改后都需要回归。

不同点：

- 普通函数常有确定输出；
- 开放式生成可能存在多个合格答案；
- AI 测试更多判断“是否满足标准”，而不是只做字符串全等；
- 一部分质量指标需要规则、模型评审或人工评审结合。
