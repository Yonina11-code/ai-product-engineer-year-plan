## 1. 大模型不是答案数据库

传统数据库查询更接近：

```text
查询条件
→ 在已有数据中匹配
→ 返回符合条件的记录
```

大模型生成更接近：

```text
读取当前上下文
→ 计算接下来各种内容出现的可能性
→ 按生成策略逐步选择内容
→ 得到完整回答
```

它不是从数据库中找到一条预先存在的“标准回答”。

因此同一个问题可能：

- 使用不同措辞；
- 强调不同重点；
- 有时简洁，有时展开；
- 有时增加用户没有提供的信息；
- 有时生成听起来合理、实际没有依据的内容。

## 2. 什么是不确定性

不确定性并不等于随机胡说。

例如下面三句话都可能是合格结果：

```text
当前需求在现有条件下存在实现困难，建议先明确需求范围。
```

```text
这个需求目前还有一些实现条件需要确认，建议先对齐需求边界。
```

```text
基于当前信息，该需求暂时难以评估，建议先补充关键约束。
```

它们表述不同，但都可能满足任务。

真正需要控制的不是“每次必须一字不差”，而是结果是否满足产品标准。

## 3. Temperature 控制什么

可以先建立这个工程直觉：

| Temperature | 常见倾向 | 适合场景 |
| --- | --- | --- |
| 较低 | 更集中、保守、相似 | 字段提取、分类、格式化 |
| 中等 | 稳定与多样之间平衡 | 对话、改写、一般问答 |
| 较高 | 更多样、更有创造性 | 创意文案、头脑风暴 |

需要特别注意：

> Temperature 不是“质量”旋钮，也不是“真实性”旋钮。

把 Temperature 调低：

- 可能减少输出差异；
- 不能保证事实正确；
- 不能修复错误 Prompt；
- 不能补齐缺失的业务上下文；
- 不能替代结果校验。

## 5. 什么是幻觉

在当前场景中，以下内容都可能属于没有依据的信息：

```text
本周五前补齐业务规则。
```

```text
我们将在下周一提供新方案。
```

```text
从当前排期来看……
```

用户并没有提供时间、承诺或排期，但模型可能生成得很自然。

幻觉最危险的地方不是语句不通顺，而是：

> 它听起来合理，让人不容易立即发现没有依据。

## 6. 不要用“好不好听”评测

职场改写可以先使用四个标准：

| 维度 | 判断问题 |
| --- | --- |
| 原意保持 | 是否仍然表达需求存在实现困难？ |
| 事实可靠 | 是否增加了原因、时间、排期或承诺？ |
| 语气匹配 | 是否专业坚定，但不过度攻击或示弱？ |
| 可直接使用 | 是否可以在真实工作场景中发送？ |

同一个输出可能：

- 语气很好，但虚构了承诺；
- 事实正确，但语气仍然攻击性强；
- 格式正确，但没有完成改写任务；
- 文案漂亮，但不适合用户选择的沟通对象。

所以 AI 评测通常是多维度的。

## 7. 和前端测试有什么相似与不同

相似点：

- 都需要明确输入和预期；
- 都要覆盖边界情况；
- 都不能只测试一次成功路径；
- 修改后都需要回归。

不同点：

- 普通函数常有确定输出；
- 开放式生成可能存在多个合格答案；
- AI 测试更多判断“是否满足标准”，而不是只做字符串全等；
- 一部分质量指标需要规则、模型评审或人工评审结合。

## 11. 学习问答记录

### Q1：我现在学到 Day2 了，我的理解是 temperature 高低会影响 AI 的自主性；Prompt 相同结果不同，是因为 AI 不同于数据库，AI 有自己的想法，所以每次生成不一样，但大概意思差不多。

直接结论：

你的理解方向是对的，但工程表达要更准确。

更准确的说法是：

> Temperature 越高，模型在生成时越敢选择不那么确定的词，所以结果更发散、更有创造性；Temperature 越低，模型越倾向选择概率最高的词，所以结果更稳定、更可控。

同一个 Prompt 多次生成结果可能不同，不是因为它在查一条固定答案，而是因为它在根据上下文概率生成内容。

需要修正的点：

- “AI 有自己的想法”可以作为直觉理解；
- 但工程上更准确地说，AI 看起来像有想法，本质上是根据 Prompt、上下文、模型参数和采样策略生成概率最高或较高概率的内容。

对应知识点：

- AI 输出不是数据库查询；
- Prompt 相同不代表输出一定完全一致；
- Temperature 控制的是输出发散程度，不是内容真实性；
- 真实业务里需要用低 temperature、结构化输出、后端校验、失败重试等方式提高可控性。

### Q2：如果 Prompt、上下文、模型参数和采样策略都一样，生成概率一样，内容就一定一样吗？

直接结论：

不一定能保证 100% 一样，但会更接近、更稳定。

如果下面这些条件都完全一致：

- Prompt 一样；
- 上下文一样；
- 模型版本一样；
- temperature 一样；
- top_p 等采样参数一样；
- 随机种子 seed 一样；
- 推理环境也完全稳定；

那么结果可能非常接近，甚至可能一样。

但是在真实 API 场景中，通常不能把它当成数据库一样的确定输出。原因包括：

1. 大模型是一步一步预测下一个 token，不是按条件查询固定记录；
2. temperature 不为 0 时，采样过程可能从多个高概率 token 中选择；
3. 模型服务底层可能存在并行计算、浮点误差、模型版本更新等变化。

可以这样记：

```text
数据库：
输入条件一样 -> 返回结果一样

AI：
输入条件一样 -> 返回结果大概率相似，但不保证完全一样
```

工程结论：

如果要做创意类功能，可以允许输出变化；如果要做分类、字段提取、风险判断等稳定任务，就不能只靠模型自然生成，而要加低 temperature、固定 Prompt 模板、JSON 结构化输出、后端字段校验、失败重试和必要的人工确认。
