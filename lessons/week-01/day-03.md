# 第3课：Prompt 在系统中的作用

建议时间：45～60 分钟  
前置条件：完成三次模型输出观察

## 1. Prompt 不是一句神奇咒语

Prompt 的作用是把产品需求转换成模型能够执行的任务说明。

一个实用的 Prompt 通常包含：

```text
角色或能力范围
任务
必要上下文
约束
输入
输出格式
示例（必要时）
```

并不是每次都要把这些部分全部写得很长。关键是让模型知道：

- 要做什么；
- 不能做什么；
- 什么结果算合格。

## 2. 从模糊要求到可检查要求

模糊版本：

```text
请说得好听一点。
```

问题：

- “好听”没有明确标准；
- 模型可能过度道歉；
- 模型可能改变用户立场；
- 无法稳定评测结果。

改进版本：

```text
将用户原话改写为适合发给同事的工作消息。

要求：
1. 保留原始事实和立场。
2. 语气专业、直接，但不带攻击性。
3. 不增加用户没有提供的原因、时间和承诺。
4. 控制在 80 字以内。
```

这些要求更容易测试。

## 3. 第一次 Prompt 改进练习

根据第2课发现的问题，给原始任务增加三条约束。

格式：

```text
任务：

约束：
1.
2.
3.

用户输入：
```

然后重新测试一次。

记录：

- 哪个问题改善了；
- 哪个问题仍然存在；
- 是否产生了新的问题。

## 4. 不要一次修改太多

如果一次同时增加十条约束、三个示例并更换模型，即使结果变好，你也不知道是哪项修改起作用。

正确的实验习惯是：

1. 先记录旧版本结果；
2. 明确一个问题；
3. 只修改少量内容；
4. 用相同测试重新验证；
5. 记录改善和退化。

这和排查前端问题很像：控制变量，比“凭感觉重写”更可靠。

## 5. 今日验收

1. Prompt 和用户输入是什么关系？
2. 为什么“说得好一点”不是一个容易评测的要求？
3. 为什么一次不应该修改太多 Prompt 内容？
4. 你新增的三条约束分别解决什么风险？
